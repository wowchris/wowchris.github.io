PCRE源码安装使用

发表于 2017-02-13 | 分类于 linux

最新版本是pcre2，但是在兼容性上存在问题，这里选择了8.39版本，安装通过。

1、下载地址：http://www.pcre.org/
pcre-8.39.tar.gz

2、解压缩：
tar xvf pcre-8.39.tar.gz

3、配置： 

cd pcre-8.39
./configure --prefix=/usr/local/pcre-8.39 --libdir=/usr/local/lib/pcre --includedir=/usr/local/include/pcre
configure有许多参数可配，具体参见./configure --help及手册

4、编译： 

make

5、安装：
make install

6、检查： 

ls /usr/local 检查是否有pcre-8.39目录
ls /usr/local/lib   检查是否有pcre目录
ls /usr/local/include   检查是否有pcre目录

7、将库文件导入cache： 

方法1：在/etc/ld.so.conf中加入： ／usr/local/lib/pcre，然后运行ldconfig
方法2：在/etc/ld.so.conf.d/下新生成一个文件（或在其中的文件中加入同样内容），文件内容为：
          /usr/local/lib/pcre，然后运行ldconfig

8、使用： 

使用pcre编写C或C＋＋程序，然后编译。
对于C程序，编译命令为：gcc -I/usr/local/include/pcre -L/usr/local/lib/pcre -lpcre file.c
对于C程序，编译命令为：gcc -I/usr/local/include/pcre -L/usr/local/lib/pcre -lpcrecpp file.cpp

centos6内核升级

发表于 2017-01-17 | 分类于 linux ， docker

 为什么要升级？ 因为我要用docker，centos7版本已经使用了新内核版本3.10.0-327.el7.x86_64但是6版本内核还是2.6.32-431.23.3.el6.x86_64,如果你不想在centos7上应用docker，那么乖乖的升级centos6的内核吧，原因就是docker需要3.8以上版本才能正常安装。

内核版本检查

[root@iZ2595dntnqZ ~]# uname -r
2.6.32-431.23.3.el6.x86_64

public key

rpm --import https://www.elrepo.org/RPM-GPG-KEY-elrepo.org

安装ELRepo

rpm -Uvh http://www.elrepo.org/elrepo-release-6-6.el6.elrepo.noarch.rpm

kernel说明

这里要说下，linux内核版本标记解释，先看下现在的内核版本

mainline

当前官网的最新版本，周期性的在整合开发者贡献后发布新版本， 注意：新版本不一定就是能用的好版本！

stable

自然是稳定版了，mainline成熟时的版本，当被标记为EOL（End of line）时则表示该版本以后出现的BUG和漏洞不在被更新维护

longterm

长期支持维护的版本，即Long Term Support，随时间推移当内核不在受到支持了也会被标记为EOL

linux-next

周期内生成的快照模式，为linux开发者用做测试

安装longterm版本

yum --enablerepo=elrepo-kernel install kernel-lt -y

编辑引导文件，修改顺序
vim /etc/grub.conf

新安装的内核会排在第一位，即0位置，更改default值为0即可

reboot重启

查看内核版本

参考资料：http://elrepo.org/tiki/tiki-index.php

docker安装和命令

发表于 2017-01-17 | 分类于 docker

docker定义

先看下docker的logo

这个图标很形象：
鲸鱼代表了docker这个应用程序，上面的集装箱则代表了docker所承载的内容，每个箱子都是一个容器container，里面装的可以是系统镜像，也可以是应用程序，那么整体上看就是一个docker里可以应用很多个service.

容器式的虚拟机，虚拟机式的容器

查了查网上的各种说法，仍然说不出个所以然，后来在dockerone上看到一篇文章，感觉和我所理解的的很贴近，关于容器与虚拟机的差异，具有普遍共识的特征归纳起来大致有以下几点：

  • 容器实例与主机共享操作系统内核，通过内核提供的运行时隔离能力为服务提供独立的用户域、文件系统、网络以及进程运行空间。虚拟机的每个实例自带操作系统，因而是一种硬件级的虚拟化隔离。
  • 容器通常是专用于运行特定服务的，它的镜像通常只包含运行该服务所需的上下文内容，许多广泛使用的镜像都只有几十MB，甚至几MB大小。虚拟机则需要提供包括内核在内的通用进程运行环境，它的镜像偏向于大而完整的全功能集合，即使一个最小的精简镜像的体积也有几百MB。
  • 容器的使用方式倾向于开箱即用，镜像提供的是一个『不可变的基础设施环境』。虚拟机则倾向于让用户根据所用的系统，自定义初始化操作，使用Ansible、Puppet这样的配置工具来进行基础设施的管理。
  • 容器在启动速度和运行性能上更有优势，虚拟机在安全性上更有优势。

通过这个介绍可以大致说明了虚拟机和容器的区别，类似于apache和nginx的讨论，实际应用看你怎么选了。

运行在硬件上的Docker：Hyper

安装

 安装官方的说明文档，需要内核版本3.8以上，所以先需要升级内核
我使用是是centos6.5，其他系统可参照官方说明安装https://docs.docker.com/engine/installation/linux/

内核升级

rpm --import https://www.elrepo.org/RPM-GPG-KEY-elrepo.org
rpm -ivh http://www.elrepo.org/elrepo-release-6-5.el6.elrepo.noarch.rpm
yum -y --enablerepo=elrepo-kernel install kernel-lt

编辑grub.conf文件，修改引导区

vi /etc/grub.conf

将default=1修改为default=0

最后，通过reboot命令重启操作系统 

[root@localhost ~]# uname -r  
3.10.104-1.el6.elrepo.x86_64

docker.io 和 docker-engine

软件是一样的，区别是:

docker.io is maintained by Ubuntu
docker-engine is maintained by Docker

cnetos6 and centos7

centos6
按照wiki的说法，需要安装EPEL才行，这样安装的docker是1.5版本，docker官方对centos6的支持停留在1.7.1版本，所以不要无谓的升级版本，升级了你反而可能不能用了，出现各种各样的问题，所以在安装docker之前先确定你的需求。
centos7
官网上有详细的安装教程，因为centos7拥有运行docker的高版本内核

centos6安装

rpm -Uvh http://download.fedoraproject.org/pub/epel/6/i386/epel-release-6-8.noarch.rpm  
yum -y install docker-io

centos7安装

准备工作：
yum install device-mapper-event-libs device-mapper device-mapper-event device-mapper-libs 

sudo tee /etc/yum.repos.d/docker.repo <<-'EOF'
[dockerrepo]
name=Docker Repository
baseurl=https://yum.dockerproject.org/repo/main/centos/$releasever/
enabled=1
gpgcheck=1
gpgkey=https://yum.dockerproject.org/gpg
EOF
注意：此方法也适用于centos6版本，且版本为1.7.1
$releasever需要改成对应的版本号，6或者7，不然会提示源错误

命令行 

yum install docker-engine
service docker start

docker info
docker run  hello-world

如果看不到信息那可能是相关软件没有更新，执行yum update吧

获取最新镜像

docker run

本地镜像：
docker images
启动容器
  准备建立个jenkins应用 

docker run -i -t -v /usr/local/src/soft/:/opt/software/ 镜像ID或仓库名称 /bin/bash

  命令解释：  
  -i,"交互模式"启动；  
  -t，启动后进入命令行模式；  
  -v，前面为主机目录，后面为容器目录；  
  /bin/bash,进入命令行

  之后你可以当这个容器是虚拟机，jenkins的安装过程这里不做描述，当配置工作完成后进行保存操作：
  docker commit <container> [repo:tag]    #repo:tag可选  

  最终运行：  
  docker run -d -t -v /usr/local/src/vote_prod:/opt/software -p 8080:3000 --name vote_prod deb-node1.0 bash -c "source /root/.profile; /usr/local/bin/forever /opt/software/vote/bin/www"
强调一下，如果想进入当前运行的容器：
该命令在退出当前容器后不退出后台容器
 
docker exec -it <新容器ID> /bin/bash

Hexo+github搭建自己的BLOG

发表于 2017-01-13 | 分类于 hexo

写在前面：
   刚接触静态网页blog不久，在网上查了查，基本上有两种：jekyll和hexo，两者都支持markdown语法，可以在github上托管自己的网站，前者不被推荐应用在windows上，并且没有没有本地服务器，无法实现预览，而hexo刚好能满足上面的要求，并且简约的next主题我非常喜欢，所以尝试着应用了hexo，感觉不错，留作备用。

环境配置

  • 安装git
    地址 https://git-scm.com/download/
  • 安装node.JS
    下载：http://nodejs.org/download/

一般情况下，选择默认安装就可以了，安装之后可检查一下是否安装正确

基本上环境已经建好了

hexo安装配置

先介绍一下常用命令

$ hexo n == hexo new#创建新文章
$ hexo g == hexo generate   #生成静态文件
$ hexo s == hexo server #本地服务器启动，可预览页面
$ hexo d == hexo deploy #本地文件发布至github

安装命令行：

本地新建个文件夹hexo（自命名），之后右键选择Git Bash Here打开git命令窗口，依次执行如下命令： 

hexo init
npm install
hexo clean
hexo g 
hexo s
执行完上面的命令，就可以访问本地站点了http://localhost:4000/

hexo的git扩展插件
npm install hexo-deployer-git --save

显示如下页面

hexo文件夹内容包括： 

hexo/
  |- node_modules/  # hexo需要的模块，不需要上传GitHub  
  |- themes/# 主题文件，需要上传GitHub的dev分支  
  |- sources/   # 博文md文件，需要上传GitHub的dev分支  
  |- public/# 生成的静态页面，由hexo deploy自动上传到gh-page分支  
  |- package.json   # 记录hexo需要的包信息，不需要上传GitHub  
  |- _config.yml# 全局配置文件，需要上传GitHub的dev分支  
  |- .gitignore # hexo生成默认的.gitignore，它已经配置好了不需要上传的hexo文件

github部署配置

登陆github，注册一个用户，创建一个专门用来放置博客的仓库，Repository name命名格式为your_name.github.io，我的是wowchris.github.io,这样github仓库就算建好了，但是还没有内容。

接下来配置ssh_key,在之前的终端里输入：

ssh-keygen -t rsa -C "Github的注册邮箱地址"

几次enter之后，会生成两个文件id_rsa和id_rsa.pub，用notepad或者ultraedit之类的编辑器打开id_rsa.pub，复制里面的内容，打开你的github：https://github.com/settings/keys,新建一个ssh key，Title自己定义，将刚刚复制的内容粘贴到Key输入框，点击Add SSH key即可。

ssh -T git@github.com

 该命令会判断你的SSH是否配置好，如果出现 You’ve successfully authenticated的语句，证明SSH配置成功。

配置BLOG域名

当然了，前提是你要有自己的域名，比如我的：www.jjl1979.com,之后在github上你的博客仓库里点击create a new file here, 在github.io后面跟CNAME,edit file内容为你的域名

确认提交后点击download zip下载，解压里面的CNAME文件到hexo的source目录下，之后登陆域名网站配置你的域名，我用的是阿里域名

github的主机地址： 

$192.30.252.153  
$192.30.252.154

设置域名解析

几分钟后，域名就可以正常访问了。

hexo主题-next

搜了搜hexo的主题，最后选了hexo，原因就是因为它简洁，干净。

首先cd到你的hexo目录
clone Next代码并存放到 themes/next 目录下

git clone https://github.com/iissnan/hexo-theme-next themes/next

修改配置文件_config.yml

theme: next

这里有一点值得注意，hexo的配置文件在开头部分，如果应用了next主题，那么配置文件要改成这样：

language: zh-Hans
timezone: Asia/Shanghai

n每个主题都有自己的独立配置文件，路径为 your_hexo/thems/next，详细内容可参见next文档说明

相关插件

1. hexo-asset-image-本地图片处理
确认_config.yml中状态post_asset_folder: true
在git bash中执行： 

npm install https://github.com/CodeFalling/hexo-asset-image --save

在hexo/source/_posts下创建一个和md文件同名的文件夹，放进需要的图片，然后在md中插入： 

![](目录名/文件名.png)

当图片路径配置完成后执行clean操作，避免出现路径错误情况 

$ hexo clean
$ hexo generate
$ hexo server

2. feed插件-RSS

npm install hexo-generator-feed

修改_config.yml，增加以下内容

# Extensions
Plugins:
- hexo-generator-feed

#Feed Atom
feed:
  type: atom
  path: atom.xml
  limit: 20

3. hexo-addlink-永久链接
执行git bash进入你的blog目录

$ npm install hexo-addlink --save

配置_config.yml文件，添加：

addlink:
  before_text: hello # text before the post link
  after_text: bye # text after the post link

之后运行命令

$ hexo clean && hexo g

4. hexo-generator-search-本地搜索

执行命令：

npm install hexo-generator-search --save

修改_config.yml文件，添加：

search:
  path: search.xml
  field: post

5. google-code-prettify-代码高亮
这里借鉴了jmblog的google-code-prettify，npm安装即可。
